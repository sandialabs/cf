Credibility Evidence Folder Structure

The credibility evidence folder structure helps the analyst to classify evidence and documents:

|-- 1-Analysis_Model
| |-- 0-Geometry CAD models, STEP files
| |-- 1-Meshing Mesh generation scripts, mesh instances generated at different resolution levels
| |-- 2-Materials Material models references and analysis code input snippets (the latter may be SAW server links to a program material repository)
| |-- 3-Computational_Models Analysis code main input file and include files
| |-- 4-Postprocessing Python or other tools for QoI evaluation, plotting results visualization, etc.
| |-- 5-Analysis_Workflow Analysis workflow to run a single instance of the model. No studies/ensembles.
|-- 2-Studies Various ensembles or single runs that don't fit into credibility buckets. Optimization, design studies, etc.
|-- 3-Credibility
| |-- 0-Documents
| |-- |-- 0-References
| |-- 1-PCMM
| | |-- 1-Code_Verification Feature coverage tool results and discussion
| | |-- 2-Physics_and_Material_Fidelity What physics models are we using, provenance?
| | |-- 3-Representation_and_Geometric_Fidelity CAD-FEM comparison, is geometry adequate? Why?
| | |-- 4-Solution_Verification Studies for parallel consistency and scalability, mesh refinement, solver controls, deterministic sensitivity etc.
| | |-- 5-Validation
| | | |-- 0-PhysSim Test process description, experimental credibility, test data
| | | |-- 1-CompSim Models of the tests, test-model comparison
| | |-- 6-Uncertainty_Quantification Uncertainty inventory, studies for uncertainty informed sensitivity, LHS or other UQ studies
| |-- 2-System_Requirements-Verification Are we meeting the requirements? Why? (Basically final design review slide package)
| |-- 3-Peer_Reviews Review panel memos and ModSim team responses.
